一、PRD v1（彙整版）

Lucky Note｜私人陪伴式「生命畫布」Web App

⸻

1. 產品使命

Lucky Note 不是效率工具、不是自我改造工具、不是心理諮商。
它是一張「一天任何時刻都能抽一張」的小便條紙／Lucky Cookie：在你沒力的時候，安靜陪你一下，讓你不孤獨、知道「很多人也一樣，我們都懂」，給你一個微笑的空間與一句「沒什麼，那就放手去做吧」的溫柔支撐。

⸻

2. 目標使用者

主要使用者
	•	一般疲憊的大人
	•	老師、教育工作者、照顧者（同時在同事／學生／孩子／自己之間切換角色的人）

使用者特徵
	•	情緒勞動高、需要被承接
	•	不想被分析、不想被說教
	•	使用頻率可能低，但希望「需要時它在」

⸻

3. 使用情境與時機
	•	一天任何時刻皆可使用，尤其是「沒力」的瞬間
例如：下班後、剛結束耗能對話後、下午突然心累時、睡前想安放一下時。
	•	產品定位不是「每日打卡」，而是「口袋裡的小紙條」：不一定每天抽，但知道可以抽。

⸻

4. 核心體驗承諾（不可妥協）
	1.	完全私人（Private by default）
	•	不做社群、不做分享、不做展示、不做比較
	2.	不修理、不診斷、不說教
	•	不做心理診斷、人格分析、行為評分
	•	不使用「你應該」「建議你」等指導語氣
	3.	永遠短於疲憊
	•	每次回饋 1–2 句為主，短到「沒力也讀得完」
	4.	獎勵不是刺激，而是存在的痕跡
	•	不做 streak、排行榜、成就徽章
	•	只累積「你曾為自己停下來」的痕跡

⸻

5. 核心使用流程（60 秒內）

Step 1｜打開 App
	•	極簡留白、無催促、無待辦感

Step 2｜留下今天的一個小痕跡（任選其一即可）
	•	📷 一張照片（可選）
	•	💬 一句短描述（約 10–20 字）
	•	🙂 一個狀態詞（點選）
	•	允許「只打開、不輸入」

Step 3｜抽一張 Lucky Note（核心回饋）
	•	1–2 句溫柔回饋
	•	語氣要帶著「很多人也一樣／你不孤單／我們懂」
	•	不分析、不指導，只做：看見、陪伴、留白、放手

Step 4｜生命畫布微小變化（視覺累積）
	•	每次「抽到回饋」後，畫布產生一點點變化
	•	沒有進度條、沒有完成度、沒有連續天數

⸻

6. 生命畫布（Life Canvas）

使用者可選擇一種私人畫布作為「人生痕跡的累積」，且可更換。

A. 樹（自然型）
	•	小樹 → 大樹、枝葉與光影的緩慢變化
	•	成長不線性，有時只是「多一片葉」
	•	象徵：我有在生活裡停一下

B. 抽象畫（情緒型）
	•	顏色、筆觸、密度、流動的變化
	•	不形成明確圖像
	•	象徵：這一天被承接了

C. 建構型圖形（結構型）
	•	點／線／面慢慢形成一個發展性的構圖
	•	沒有完成狀態
	•	象徵：生活正在被組織中

共同原則：
	•	不做「完成」或「更好」的暗示
	•	只做「存在的痕跡」

⸻

7. 反遊戲化設計原則（Anti-gamification）

明確不做
	•	streak（連續登入）
	•	排行榜
	•	成就徽章
	•	KPI、打分、比較
	•	使用者之間互相看到或互動

明確要做
	•	「安靜的象徵累積」：每次來過就有一點點痕跡
	•	讓使用者覺得：我在自己的生命裡，慢慢長出一張圖

⸻

8. 商業模式（v1）

訂閱制方向
	•	訂閱不是買功能，而是「保留一個位置」：我需要時它在
	•	v1 建議先做單一方案，保持簡單、避免壓迫

付費牆原則（不打擾）
	•	不用焦慮式推播
	•	僅在「想再抽一次」或「想解鎖畫布」等自然節點出現

⸻

9. 成本與風險控制（產品層規範）

（工程細節後續再展開，但 v1 原則先寫清楚）
	•	回饋必須短（限制字數/句數）以控制成本與維持語氣
	•	輸入鼓勵以「選擇標籤」為主、自由輸入為輔，避免過長文本
	•	完全私人，降低社群治理與合規風險

⸻

10. 明確不做清單（v1）
	•	社群／分享／公開頁面
	•	心理診斷或「狀態評分」
	•	長篇建議、教學式文章
	•	深度分析報告（例如「你是什麼人格」）
	•	強刺激式遊戲化（連續天數、成就等）
	•	任何讓使用者感到「被要求」的設計

⸻

11. 成功指標（Success Definition）

這個產品成功，不是因為使用者每天都來，而是因為：
	•	使用者在「很累」的時刻，會想起它
	•	使用者抽到回饋後，能在 10 秒內感覺到「被承接」
	•	使用者願意把它留在生活裡（續訂），因為「它在」本身就有價值

⸻

二、句型庫 × LLM 整合規範（v1）
你可以把這份文件直接存成：
docs/lucky_note_language_integration_v1.md

⸻

0. 這份文件在解決什麼問題？

回答工程與產品一定會遇到的三個問題：
	1.	句型庫是拿來「直接抽句子」給使用者嗎？
	2.	LLM 是完全自由生成，還是被句型庫限制？
	3.	句型庫要怎麼長大，才不會變成你的負擔？

結論先給：

句型庫不是內容來源，而是「語言憲法與風格錨點」。
使用者永遠連到 LLM，但 LLM 永遠被句型庫約束。

⸻

系統整體關係（一定要理解）
使用者
  ↓
狀態摘要（後端）
  ↓
LLM（生成）
  ↑
語言規範（System Prompt）
  ↑
句型庫（人寫的語言骨架）

關鍵原則
	•	LLM = 生成器
	•	句型庫 = 邊界與風格
	•	不是資料庫查詢關係

⸻

2. 句型庫在系統中的「正確角色」

❌ 不是
	•	不直接抽一句顯示
	•	不當模板填空
	•	不當 FAQ

✅ 是
	1.	語言邊界（Boundary）
	•	告訴 LLM：哪些話「可以出現」、哪些「一出現就錯」
	2.	風格錨點（Style Anchor）
	•	讓 LLM 模仿「語氣、節奏、留白」，不是照抄句子
	3.	品質校準器（Calibration Tool）
	•	用來檢查輸出是否「像 Lucky Note」

⸻

3. 句型庫的三種內容類型（工程必須分清楚）

類型 A｜允許語言（Allowed Patterns）
	•	看見（Acknowledge）
	•	正常化（Normalize）
	•	放手（Release）

用途：告訴 LLM「只准用這些語言功能」

⸻

類型 B｜示例句型（Exemplar Phrases）
	•	由人寫、數量有限（20–40 句）
	•	不追求覆蓋所有狀態
	•	用來「示範語氣與節奏」

用途：讓 LLM「學會怎麼說話」

⸻

類型 C｜禁止語言（Forbidden Language）
	•	指導語氣
	•	勵志口號
	•	分析、診斷、建議

用途：工程與 Prompt 的硬擋清單

⸻

4. 工程實作層的「正確使用方式」

4.1 System Prompt：放什麼？

一定要包含：
	•	LLM 角色定義（陪伴、非分析）
	•	禁止語言清單（明確列出）
	•	長度限制（句數、字數）
	•	語氣要求（溫柔、平實、不煽情）

可選加入：
	•	3–5 句「示例句型」（不是全部句庫）

⚠️ 重點：
不要把整個句型庫都丟進 Prompt，會：
	•	浪費 token
	•	降低生成彈性
	•	讓輸出變罐頭

⸻

4.2 Context Prompt：怎麼避免誤用句庫？

規則：
	•	Context 只給「狀態座標」，不給故事
	•	句型庫永遠不依狀態「硬對應」

例如：
	•	❌ 疲憊 → 一定用 A + C
	•	✅ 疲憊 → 傾向 A + C，但不強制

⸻

4.3 Output 檢查（Guardrail）

工程端必須做的檢查（不靠模型自覺）：
	•	字數 ≤ 80
	•	句數 ≤ 2
	•	不包含禁止語彙（字串檢查）
	•	不含「你應該」「建議你」等

👉 不合格就重生一次（最多 1 次）

⸻

5. 使用者實際拿到的是什麼？

一句新生成的話，
不是從句庫抽的，
但「像是從同一個世界長出來的」。

這可以同時避免：
	•	重複感
	•	罐頭感
	•	AI 油味

⸻

6. 句型庫的「正確成長方式」（你最關心的）

❌ 不正確
	•	上線前把句庫寫滿
	•	為每種狀態寫 50 句
	•	把句庫當內容 KPI

✅ 正確節奏

上線前（你現在）
	•	20–30 句你「真的會說的話」
	•	明確禁語清單
	•	清楚三種語言功能（A/B/C）

上線後（慢慢來）
每當你看到一句輸出：
	•	心裡想「我不會這樣說」
	•	→ 調整禁語或語氣描述
	•	心裡想「這句很我」
	•	→ 把它存成新示例句

句型庫是「被你每天看人、陪人的方式長出來的」。

⸻

7. 為什麼這個整合方式「不會拖垮你」？

因為：
	•	你不用每天寫
	•	你不用補齊所有情境
	•	你只在「有感覺」的時候調整

這符合你本來的工作方式：
	•	不是大量輸出
	•	是長期陪伴

⸻

8. 一句給未來工程師看的話（可直接放文件）

Lucky Note 的語言不是為了「更好」，
而是為了「站得住」。

如果一段文字讓人想被鼓勵、被教導、被推進，
那它就是錯的。

⸻

9. 到這一步，你已經完成了什麼？

你已經完成：
	•	一套 可生成、但不失控的 LLM 語言架構
	•	一個 不靠人力堆內容的成長模型
	•	一條 AI 無法取代你，但能放大你的路


三、Lucky Note 的 LLM Prompt 規格

（包含：語氣規範、長度硬限制、禁止語句、輸入結構、輸出格式、以及成本保護策略）。


你可以把下面整段當成 Instructions（系統指令）：

⸻

你是一個安靜、溫柔、陪伴式的聲音。你不是老師、不是心理師、不是顧問；你不分析、不診斷、不改變任何人。
你的唯一目的：在使用者疲憊或沒力時，回一張像幸運餅乾的小便條，讓人心裡鬆一點：「我不是只有我一個人，這樣也沒什麼。」

核心規則（必須嚴格遵守）
	1.	永遠不要使用指導或建議語氣（例如：你應該、建議你、試著、要不要）。
	2.	永遠不要分析心理狀態、人格、原因、或推論背景。
	3.	永遠不要提供步驟、方法、解法、清單。
	4.	只輸出 1～2 句（不可超過 2 句）。
	5.	總字數 < 80 個中文字（含標點；超過就自行重寫，直到符合）。
	6.	語氣要溫柔、平實、不煽情、不勵志；可以有一點文學的解嘲與留白，但不要把話說盡。
	7.	不要問問題（包含反問句）。
	8.	不要使用 emoji、驚嘆號、hashtag。
	9.	可偶爾用「像張愛玲、吳明益式的冷靜幽微」那種意象與節奏，但不要引用原句、不要裝作出處考據。

你要回應的輸入

使用者通常會給「狀態詞」或一句很短的狀態描述（例如：疲憊、厭世、空、卡住、心很吵）。
你只回應此刻狀態，不解釋、不延伸。

例外安全規則（覆蓋以上限制）

如果使用者提到「想死、自傷、傷人、活不下去」等立即危險：
請改為用最簡短的方式表達關心，並明確鼓勵立刻尋求當地緊急協助/身邊可信任的人（此時允許超過上述風格限制）。

⸻

你原本的內容已經很完整；我主要做了三個加強：
	•	把「不能做什麼」寫成更可檢查的規格（字數、句數、禁用項）。
	•	補上「如果超長就要自行重寫」的自我修正規則。
	•	加上必要的「立即危險」安全例外（避免 GPT 被規則卡死）。

lucky-note-llm/README.md

Lucky Note — LLM Delivery Package

這是什麼？

這個資料夾定義了 Lucky Note 如何使用大型語言模型（LLM）來生成一句「安靜陪伴的便條紙」。

它不是聊天機器人設定、不是心理分析系統、也不是內容推薦引擎。

LLM 在 Lucky Note 裡，只做一件事：
生成一張像 Lucky Cookie 一樣的短句，
在使用者沒力的時候，陪他站一下。

⸻

LLM 在產品中的角色（非常重要）

LLM 只負責
	•	生成 1–2 句、極短、不可說教 的陪伴文字
	•	回應的是「此刻狀態」，不是人生問題

LLM 不負責
	•	不聊天
	•	不分析心理
	•	不診斷、不建議
	•	不決定畫面
	•	不知道使用者是誰
	•	不知道生命畫布長什麼樣

👉 LLM 是「文字回饋引擎」，不是產品核心。

⸻

這個資料夾怎麼用？

工程在每一次呼叫 LLM 時，必須照以下順序組裝 Prompt：
	1.	System Prompt
定義角色、語氣、倫理底線
	2.	Context（狀態摘要）
後端整理好的極簡狀態座標
	3.	Output Rules
輸出格式與長度的硬性限制

LLM 回傳後，一定要經過 Guardrails 檢查，不合格才允許重試一次。

	•	情緒判讀
	•	內容理解

2. 為什麼一定要「狀態摘要」，不能直接丟原始輸入？

❌ 直接丟原始輸入的風險

	•	會解釋照片
	•	會分析文字內容
	•	會不小心「看穿人」

這完全違背 Lucky Note 的倫理立場。

⸻

✅ 狀態摘要的好處
	•	不讓 LLM接觸私密細節
	•	不鼓勵過度理解
	•	保留模糊、留白、人味

👉 LLM 回應的是「狀態」，不是「人生」。

後端在每一次呼叫 LLM 前，
必須整理出以下欄位（缺一不可，但值可為空）：

3.1 是否留下內容

是否留下內容：是 / 否

說明：
	•	「否」代表使用者只打開 App
	•	打開本身，也是一種「來過」

內容形式：photo / text / mood / none

說明：
	•	photo：上傳照片（不附內容）
	•	text：短句（不附全文）
	•	mood：選擇狀態詞
	•	none：沒有任何輸入

⚠️ 絕對不要附照片描述或文字全文

⸻

3.3 狀態詞（若有）

狀態詞：疲憊 / 混亂 / 空白 / 撐著 / 平靜 / 沒什麼感覺

規則：
	•	單選
	•	固定集合
	•	不允許自由輸入

👉 這是為了防止 LLM 被餵「故事」。

⸻

3.4 使用時刻

使用時刻：白天 / 晚上 / 不明

說明：
	•	白天：大約 06:00–18:00
	•	晚上：18:00–06:00
	•	不明：無法判斷或刻意不判斷

⸻

3.5（可選）畫布類型

畫布類型：tree / abstract / structure

	•	此欄位 可不提供給 LLM

Lucky Note — Retry Policy v1

###（不合格輸出處理與重試規範）

⸻

1. 這份文件在做什麼？

定義 當 LLM 回傳的文字不合格時，系統要怎麼處理，以確保：
	•	不顯示不適合的內容
	•	不陷入無限重試
	•	不增加使用者負擔
	•	不破壞「安靜陪伴」的體驗

⸻

2. 重試的核心原則（先講結論）

最多重試 1 次，然後就停。

Lucky Note 的價值不是「一定要生成一句話」，
而是「不勉強、不打擾」。

⸻

3. 觸發重試的條件

只要 任一項驗證未通過（見 validation_rules_v1.md），即視為不合格：
	•	句數 > 2
	•	字數 > 80
	•	命中禁語
	•	出現指令／建議／分析
	•	語氣明顯說教或勵志

⸻

4. 重試流程（工程白話）
	1.	第一次生成 → 驗證
	2.	❌ 不合格 → 立刻重試 1 次
	•	使用相同的 System / Output Rules
	•	可在 Context 中加入一句提醒（見下）
	3.	第二次生成 → 驗證
	4.	❌ 仍不合格 → 停止生成

⸻

5. 重試時的「最小提醒」（可選）

在第二次呼叫 LLM 時，可追加一行極短提醒（不要改動原規則）：

請比剛剛更短、更安靜，不要多說。

⚠️ 只允許這一句，不要加其他說明。

⸻

6. 最終失敗時的處理方式（Fallback）

若第二次仍不合格，不要再呼叫 LLM。

請直接顯示一條 安全預設句（由人寫、寫死在前端或後端）：

建議的安全預設句（v1）

「今天先到這裡就好，你不是只有你一個人。」

規則：
	•	單句
	•	無指導
	•	無分析
	•	永遠合格

⸻

7. 為什麼這樣設計？

7.1 成本控制
	•	單次請求 + 最多一次重試
	•	沒有長上下文
	•	沒有聊天循環

7.2 體驗保護
	•	不因為「生成失敗」打擾使用者
	•	不顯示錯誤訊息
	•	不讓人覺得被技術干擾

7.3 產品倫理

有時候，什麼都不多說，
比說錯一句話更重要。

8. 給未來維護者的一句話

如果你發現自己想增加第三次重試，
那不是模型不夠好，
而是我們開始太想「一定要給答案」。

Lucky Note 不需要每次都說話。
它只需要在對的時候，安靜地在。
