TeacherOS
AI 協作教學系統全流程測試報告

三合一報告：新手教師體驗 | 系統測試回饋 | 教育部推廣策略


測試身份：Claude 老師（八年級英文科任 + 班級導師）
測試班級：Claude 班（18 人，10 週學期）
測試範圍：Block 1-4 全流程（學季規劃 → 課堂設計 → 歷程紀錄 → 期末評量）
測試日期：2026 年 3 月 3 日
系統版本：TeacherOS Beta
報告撰寫：Claude（模擬新手教師 + 測試工程師 + 教育部負責人三重角色）
執行摘要
本報告記錄了一位完全的新手教師（Claude 老師）從零開始使用 TeacherOS 系統的完整歷程。測試涵蓋了從閱讀教師指引、建立班級資料夾、到走完 Block 1-4 全流程的每一個步驟，並實際產出了一個完整的 10 週英文課學期的所有文件。
核心發現：TeacherOS 的架構設計極為精密，能有效引導教師產出高品質的差異化教學內容。然而，新手教師在初始設定階段存在明顯的認知負荷過高問題，且部分流程文件仍預設使用者具備一定的技術背景。

產出成果一覽

第一部分：新手教師體驗紀錄
以下以第一人稱記錄一位名叫 Claude 的新手英文教師，從拿到系統到完成一整個學期工作的完整歷程。
1.1 初始設定階段（約 60-90 分鐘）
步驟一：閱讀 Teacher Guide
teacher-guide.md 是新手教師的第一份文件。全文約 500 行，涵蓋了系統簡介、環境設定、權限說明、四大工作區塊、Google Docs 輸出流程等。
新手體驗觀察：
文件結構清晰，從「這套系統能幫你做什麼」開始，是正確的起點
帳號與工具表格一目了然，但同時需要 GitHub + AI 工具 + Google Drive + Pandoc 四樣東西，對非技術背景教師來說心理壓力偏大
Git clone 指令對完全不懂終端機的老師是一個關鍵障礙點——需要額外的圖文教學或影片
「建立你的個人設定檔」段落指示清楚，但 environment.env.example 的位置需要老師自己找到，建議直接在指引中放截圖
Hook 安裝步驟簡單明瞭，但老師可能不理解「Git 防呆機制」的意義，建議用更生活化的比喻
步驟二：閱讀 AI_HANDOFF.md
這份文件是給 AI 讀的，但新手教師也需要理解系統如何運作。文件寫得極為精確，技能觸發語對照表非常實用。
新手體驗觀察：
對新手教師而言，「AI 每次對話必讀」這個概念很新——需要在 Teacher Guide 中更明確說明「你不需要手動做這些，AI 會自動完成」
Slash command 對照表（/syllabus、/lesson 等）是極佳的快速啟動入口
技能系統的設計讓不同 AI 工具都能使用同一套流程，這個可攜性非常強
步驟三：建立班級資料夾
按照 Teacher Guide 的指引，複製現有班級資料夾結構，建立 class-claude/ 並修改 project.yaml。
新手體驗觀察：
Teacher Guide 說「複製 class-9c/ 資料夾結構，再修改內容」，但 class-9c 已有大量 David 的個人資料，容易混淆
建議提供一個空白的 class-template/ 資料夾，裡面只有空的 YAML 骨架
students.yaml 的格式需要教師理解 DI 分類邏輯——建議在 Teacher Guide 中加入 DI 分類的簡易問卷或決策樹
roster.yaml 涉及學生真名，Teacher Guide 有提到 gitignore 保護，這很好
步驟四：改寫 TeacherOS.yaml
Teacher Guide 要求新手教師改寫 ai-core/teacheros.yaml，但這是系統核心設定中受 CODEOWNERS 保護的檔案。
關鍵問題：
Teacher Guide 第四節說「打開 teacheros.yaml，依照以下方向修改」，但 acl.yaml 明確規定 teacher 角色不能修改 ai-core/ 路徑——這是一個流程矛盾
解決方案已存在但未充分說明：workspace 機制允許教師在 workspaces/{teacher-id}/ 建立自己的覆寫版本
建議在 Teacher Guide 中明確說明：「你不需要修改原始的 teacheros.yaml，而是在你的 workspace 中建立自己的版本」
1.2 Block 1：學季整體規劃
作為 Claude 老師，我選擇以《Freedom Writers Diary》作為八年級英文課的核心文本。這個選擇基於華德福教育中八年級對應「革命史」的發展階段——學生內在的叛逆與對公平正義的強烈感受，與 Freedom Writers 的故事高度共鳴。
Block 1 流程體驗：
Step 1（教師輸入）：系統要求收集 8 個必要欄位的資訊——標題、班級、授課教師、教學目標、教學策略、教學規劃、學生任務、評量方式。流程清晰，引導性強
硬性前置攔截：系統要求教師提供研究文件才能開始，這是一個極為重要的品質控管機制。它迫使教師「先思考再讓 AI 幫忙」，避免了空洞的 AI 生成內容
Step 2（DI 檢核）：AI 自動檢查學習優勢覆蓋度（語言/圖像/動作/藝術）和能力動機矩陣（ABCD）。這確保了差異化教學不是事後補上，而是從規劃階段就內建
Step 3（產出）：最終產出的教學大綱為 277 行，涵蓋完整的 10 週 4 單元規劃。語言溫暖但不說教，適合家長和學生閱讀
產出品質分析：
優點：大綱具體到每週要做什麼、讀什麼、寫什麼，不是抽象的「培養能力」
優點：評量方式清楚列出百分比，合計 100%，沒有模糊空間
優點：寫作風格邊界被嚴格執行——沒有出現「帶得走的能力」等空泛修辭
改進空間：大綱主要以英文撰寫，對台灣家長閱讀可能偏難，建議提供中英對照版本的模板選項
1.3 Block 2：課堂教學設計
Block 2 是系統中工作量最大的區塊，需要為每一節 45 分鐘的課堂產出完整的教學流程。
Unit 1 詳細教案（8 節課）：
每節課都嚴格遵守四段結構：Warm-up → Main Activity → Check/Assessment → Closing
差異化安排是最大亮點——每節課都為 ABCD 四類學生提供了具體的、有畫面感的任務描述
例如 D 類學生：「給他們極度具體的任務：在文本中找出 5 個認得的英文單字，圈起來，告訴旁邊的人這些字是什麼意思。成功完成 = 一個小小的勝利」
特殊需求學生（CL-07 讀寫困難、CL-15 情緒調節）在每節課都有明確的支援方案
教師備忘欄位實用：包含材料清單、時間風險、課後觀察提示
Unit 2-4 濃縮教案（32 節課）：
採用「單元概覽 + 每週摘要 + 關鍵差異化活動 + 評量檢核點」的濃縮格式
保留了差異化的核心精神，同時減少了重複性的格式化工作
建議：系統可以提供「詳細版」和「濃縮版」兩種模板讓教師選擇
1.4 Block 3：教學歷程紀錄
Block 3 的設計理念是「消化過去發生的課堂」——教師只需要用語音或文字傾倒課後印象，AI 負責結構化拆解與歸檔。
三維紀錄庫的實際產出：
學生歷程檔：為 6 位代表性學生（每種 DI 類型各一 + 兩位特殊生）建立了跨越 10 週的觀察紀錄，每位 4-5 筆帶日期標記的條目
班級教學歷程：按單元記錄了班級整體動力、成功活動、需調整活動、常見困難
教師反思日誌：6 篇帶日期的反思，展現了教學觀點的演變——從初期的「驚訝與謙卑」到後期的「重新定義成功」
品質亮點：紀錄語氣真實溫暖，像一位真正的老師在寫觀察筆記，而非 AI 式的分類報告
1.5 Block 4：學習評量與結案
Block 4 是整個學期的收成，將日常碎片化的紀錄彙整為正式文件。
教師教學歷程總檔：
完整回顧了 10 週教學旅程，分析了每個單元的成效
包含 DI 系統有效性分析——哪些差異化策略真正有用、哪些需要改進
跨學科連結的自然浮現被記錄下來（歷史、視覺藝術、社會）
學生期末評量總檔：
全班歷程摘要約 200 字，適合家長閱讀
18 位學生每人都有獨立的三段式評量：質性描述 + 大綱檢核 + 量化成績
品質亮點：每位學生的質性描述都是獨特的，引用了具體的課堂場景和作品內容
成績分布合理：A 類學生 82-98 分，B 類 70-87 分，C 類 68-78 分，D 類 54-67 分
系統自動從 roster.yaml 反查真實姓名寫入總檔，方便列印給家長
第二部分：系統測試回饋
以測試工程師視角，對 TeacherOS 系統架構、產出品質、使用流程提出分析與改進建議。
2.1 系統架構分析
母架構評價：設計精良
TeacherOS 的三層記憶模型（Layer 1 身份層 / Layer 2 專案層 / Layer 3 工作線層）是整個系統的核心優勢。這個設計讓 AI 能在任何時間點、任何工具上接手工作，而不需要教師重新說明背景。

2.2 發現的問題與改進建議
問題 1：初始設定的認知負荷過高（嚴重度：高）
新手教師需要同時處理 GitHub 帳號、終端機操作、Git clone、環境設定、Hook 安裝等技術任務。即使每個步驟都有說明，累積起來的心理壓力足以讓非技術背景的教師放棄。
建議改進：
提供「一鍵安裝腳本」：bash setup/quick-start.sh，自動完成 clone、環境設定、hook 安裝
製作 5 分鐘的影片教學，搭配螢幕截圖
提供「技術支援伙伴」制度：每位新手教師配一位已熟練的同事
問題 2：TeacherOS.yaml 修改路徑矛盾（嚴重度：高）
Teacher Guide 要求教師修改 ai-core/teacheros.yaml，但 ACL 規定 teacher 角色不能修改 ai-core/ 路徑。Workspace 覆寫機制已存在但文件中未充分說明。
建議改進：
在 Teacher Guide 中明確指出：「不要直接修改 ai-core/teacheros.yaml」
提供 workspaces/_template/ 中的 teacheros-personal.yaml 範本
AI 在偵測到新使用者時，自動引導建立個人 workspace
問題 3：缺少空白班級模板（嚴重度：中）
Teacher Guide 建議複製 class-9c/ 資料夾，但 class-9c 已有大量 David 的個人資料（22 名學生的 DI 分類、多版本大綱等），新手教師不知道哪些該保留、哪些該刪除。
建議改進：
建立 projects/_class-template/ 空白模板資料夾，只含骨架 YAML
每個 YAML 中加入註解範例，教師填空即可
提供 DI 分類決策樹或問卷，幫助教師為學生分類
問題 4：Block 2 工作量巨大且缺乏彈性選項（嚴重度：中）
為 40 節課（10 週）每節產出完整的四段式教案，對任何教師來說都是巨大的工作量。系統目前只有一種詳細度。
建議改進：
提供三種詳細度選項：精簡版（單元概覽）、標準版（每週摘要+關鍵活動）、完整版（逐節教案）
允許教師對第一單元做完整版，後續單元自選詳細度
建立「教案模板庫」——累積成功的教案成為可重用資源
問題 5：Block 3 缺乏語音輸入的具體操作指引（嚴重度：低）
Block 3 設計理念是教師用語音傾倒課後印象，但 Teacher Guide 未說明如何實際操作語音輸入（用什麼 app？錄音檔如何傳給 AI？）。
建議改進：
提供語音輸入的工具推薦（如 Claude Code 的語音模式、手機錄音 + 上傳）
給出語音輸入的模板句型：「今天第三節課，CL-05 的表現是...」
問題 6：雙路徑儲存增加維護複雜度（嚴重度：低）
每份產出同時存入 content/english/ 和 english/content/ 兩個路徑。設計初衷是提供班級視角和科目視角，但增加了出錯風險。
建議改進：
考慮使用符號連結（symlink）替代實體複製，確保一致性
或統一為單一路徑 + 索引檔案

2.4 對母架構的強化建議
建立「新手引導精靈」：AI 偵測到新使用者時，自動進入引導模式，一步步帶教師完成設定
開發「成果展示頁」：讓新教師能在開始前看到完整的產出範例（如本次測試的 class-claude 全部檔案）
建立「品質基準線」：將 David 的 class-9c 產出標記為品質範例，新教師可以對照參考
設計「漸進式導入」路徑：第一個月只做 Block 1，第二個月加入 Block 2，逐步增加，不要求一次全部上線
新增「教師社群」機制：讓不同教師的 workspace 能有選擇性地分享成功教案
第三部分：教育部推廣策略建議
以「AI 協作精進學校教育的台灣教育部負責人」視角，基於本次系統測試的完整結果，提出推廣策略與政策建議。
3.1 系統價值評估
TeacherOS 展現了 AI 協作教學系統的三個核心價值：
品質提升：差異化教學從理念變成可操作的系統化流程。教師不再需要憑經驗「猜」不同學生需要什麼，系統強制要求每一份教案都包含 ABCD 四類學生的具體安排
效率提升：一位教師在 AI 協助下，能在數小時內產出過去需要數週才能完成的完整學期教案——包含 40 節課的差異化設計
知識傳承：系統的三層記憶模型讓教師的教學經驗不再只存在於個人腦中，而是結構化地保存在 YAML 和 Markdown 檔案中，可以被未來的教師和 AI 繼承
3.2 推廣可行性分析

3.3 三階段推廣路線圖
第一階段：試點期（6 個月）
目標：在 3-5 所實驗學校驗證系統可行性
選擇 3-5 所不同類型的學校（華德福、實驗教育、體制內）作為試點
每校選拔 2-3 位「種子教師」，提供密集培訓（含技術支援）
種子教師以英文課為起點，走完 Block 1-2 流程
收集使用數據與教師回饋，修正系統問題
預算估計：每校 NT$50 萬（含 AI API 費用、培訓費、技術支援）
第二階段：擴展期（12 個月）
目標：擴展至 20 所學校，發展多科目模板
基於試點回饋，開發 Web 介面版本（降低技術門檻）
發展國文、數學、社會等科目的 DI 模板
建立跨校教師社群，分享成功教案與最佳實踐
制定「AI 協作教學」教師認證制度
預算估計：NT$2,000 萬（含系統開發、培訓、認證制度建立）
第三階段：常態化（24 個月）
目標：將 AI 協作教學納入教師專業發展體系
將 TeacherOS 類系統納入師資培育課程
建立全國性的 DI 教案資料庫，鼓勵教師貢獻與共享
發展學生學習歷程的跨學期、跨年級追蹤機制
定期評估 AI 協作對學生學習成效的影響
預算估計：NT$5,000 萬/年（含系統維運、教師培訓、研究評估）
3.4 政策建議
制定《AI 協作教學指引》：明確 AI 在教學中的角色定位——AI 處理結構化設計工作，教師專注於與學生的人際互動。TeacherOS 的「以技術自動化換取純粹低科技人文互動」哲學值得作為政策基調
建立 AI 教育資料保護框架：學生 DI 分類、學習歷程等資料的收集、儲存、分享需要明確的法律框架。建議參考歐盟 GDPR 的教育資料保護條款
設立 AI 教育研究基金：鼓勵大學與中小學合作，研究 AI 協作教學對不同年齡段、不同科目學生的實際影響
避免「一刀切」推廣：TeacherOS 的 workspace 機制允許每位教師有自己的教學風格，政策應保護這種彈性，不要變成另一套標準化課綱
重視教師的「意義架構師」角色：AI 能生成教案，但教師對學生發展階段的理解、對課堂氛圍的敏感度、對個別學生的關懷，是 AI 永遠無法替代的。政策應強化教師的專業判斷力，而非讓教師成為 AI 輸出的執行者
