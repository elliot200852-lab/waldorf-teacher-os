Chapter 6 — 系統演化史

從一個人的備課筆記
到多人協作的教學操作系統

七個版本、四十天。每一次重構都是對「教師真正需要什麼」的更深回答。
這不是軟體開發史，這是一位華德福教師學會駕馭技術的意識演化歷程。

版本演化時間軸

每個版本都解決了一個真實的教學痛點。從最初的個人備課記憶，到今日支援多位教師同時工作的完整系統——演化的動力始終是教室裡的真實需求。

v1 — 記憶的起點：個人備課 YAML +

核心問題

每次開啟新的 AI 對話，教師都必須從零開始解釋自己是誰、教什麼、學生在哪個階段。AI 是一個失憶的天才——能力超群但永遠不記得你。

解決方案

建立第一份 teacheros.yaml——一個隨時可以餵給任何 AI 的教師身份檔案。包含教師姓名、教學科目、華德福教育哲學的核心原則、學生的發展階段描述。

意義

這是「意義架構師」概念的萌芽。教師第一次意識到：我不需要學程式，我需要學會告訴機器「我是誰」。定義自己的身份，就是定義 AI 的行為邊界。

v2 — 結構的萌芽：三層記憶模型 +

核心問題

單一檔案越來越龐大。教育哲學、班級資料、備課進度全部混在一起，AI 讀取時既慢又混亂。需要一個層次分明的記憶結構。

解決方案

將單一 YAML 拆分為三層記憶架構：

層次內容更新頻率
系統核心層華德福教育哲學、發展心理學、AI 使命宣言極少變動
專案框架層差異化教學規則、品質標準、模板學期初設定
工作進度層session.yaml 進度錨點、備課內容每次對話更新

意義

教師開始像系統架構師一樣思考：哪些知識是永恆的？哪些是會變動的？如何讓 AI 在最短時間內進入工作狀態？這是教育工作者第一次有意識地設計「認知載入協議」。

v3 — 差異化的覺醒：DI 雙軸框架 +

核心問題

AI 產出的教案漂亮但「平均」——每個學生拿到一樣的內容。華德福教育的核心是看見每一個獨特的孩子，而非生產標準化教案。

解決方案

建立差異化教學（DI）雙軸框架，成為所有課程設計的強制性錨點：

軸一：學習風格偏好——語言型、視覺型、動覺型、藝術型四種路徑，每堂課至少覆蓋兩種。

軸二：能力 × 動機矩陣（ABCD）——A 組（高能力高動機：深化挑戰）、B 組（高能力低動機：點燃熱情）、C 組（低能力高動機：鷹架支持）、D 組（低能力低動機：先建立安全感）。

意義

這是系統從「幫教師產出教案」躍升為「幫教師看見學生」的轉折點。AI 不再是打字機，而是差異化教學的設計夥伴。品質標準不再是「好不好看」，而是「有沒有照顧到每一個孩子」。

v4 — 進度的錨點：Session 協議與技能系統 +

核心問題

教師在不同 AI 工具之間切換（Claude Code、Gemini），每次都要重新說明「上次做到哪了」。多個 AI 無法接力工作。

解決方案

建立 session.yaml 作為進度錨點——每次對話結束前，AI 必須更新當前的 Block、Step、下一步行動。下一個 AI 開啟時，讀取 session.yaml 即可從上次停止處繼續。

同時建立技能系統（Skills）：將常見工作流程封裝為標準化的 Markdown 規格檔。教師說「進入備課」，AI 自動讀取 lesson.md 技能檔，按步驟執行。不同 AI 工具讀取同一份技能檔，產出行為一致。

意義

系統第一次具備了「工具無關性」——技能正本是純 Markdown，任何有檔案讀取能力的 AI 都能執行。教師的智慧被編碼在文字檔中，不被任何特定 AI 平台綁架。這是真正的數位自主權。

v5 — 品質的尺度：輸出標準與模板系統 +

核心問題

AI 產出的教案品質不穩定——有時出色，有時遺漏關鍵的差異化設計。教師需要一把明確的尺，衡量每份產出是否達標。

解決方案

建立五條品質標準規則、英文課 Block 1-4 教案模板、導師業務模板。每份 AI 產出都必須通過品質自檢清單：差異化雙軸是否覆蓋？學習風格路徑是否至少兩種？ABCD 各組是否有對應的教學策略？

同時建立 build.sh 輸出腳本，一鍵將 Markdown 教案轉換為可列印的 PDF 文件，含學校 Logo 與華德福水彩浮水印。

意義

品質不再靠運氣。模板系統確保每位教師站在同一個起跑線上，而品質標準確保終點線的位置不會降低。AI 的角色從「自由發揮」被框定為「在明確規範內發揮創意」。

v6 — 空間的隔離：Workspace 獨立工作區 +

核心問題

系統開始準備迎接第二位教師。但如果兩位教師的備課資料混在一起，修改一方可能破壞另一方的工作。需要隔離機制。

解決方案

建立 Workspace（工作空間）機制。每位教師擁有完全獨立的工作區：

共讀層（所有教師共用）：教育哲學根基、差異化框架、技能正本。教師只讀不改，由管理者統一維護。

私有層（教師完全自主）：個人身份檔、班級資料、備課內容、進度錨點。100% 屬於教師個人。

同時建立 ACL（存取控制清單），明確定義每位教師的可操作路徑範圍。

意義

這是系統從「一個人的工具」轉變為「組織的基礎設施」的關鍵時刻。共讀層確保所有教師站在同一個華德福哲學基礎上；私有層確保每位教師的創造自由不被干擾。就像華德福學校本身——共同的教育理念，獨立的教學藝術。

v7 — 協作的誕生：Git 多人工作流程（當前版本） +

核心問題

Workspace 隔離了資料，但教師之間還無法安全地「分享」與「合併」工作。需要一個機制讓教師能在自己的空間自由工作，同時能在準備好時將成果合併回共用系統。

解決方案

採用 Collaborator + Branch 協作模式：

每位教師擁有個人分支（branch），命名規則為 workspace/{Teacher_ID}。教師在個人分支上自由存檔，不影響主系統。當工作準備好時，透過合併申請（Pull Request）提交審核，由管理者確認後合併回主系統。

三層保護機制

層次機制強度說明
第一層Pre-commit Hook軟保護存檔前自動檢查，提醒教師是否操作了不在權限範圍內的檔案
第二層CODEOWNERS硬保護核心檔案修改必須經過指定審核者同意
第三層Branch Protection硬保護主系統分支受保護，只有通過審核的合併申請才能進入

新增技能

save（存檔）——教師說「存檔」或「幫我存」，AI 自動執行版本控制的存檔流程（git add → commit → push）。教師完全不需要理解 Git 命令。

pull-request（合併申請）——教師說「發 PR」或「送回主系統」，AI 引導完成合併申請的建立，包含工作摘要與變更清單。

操作介面全面升級

教師手冊從技術文件重寫為 AI 口語操作介面。所有操作都提供兩種路徑：「方式 A：跟 AI 說」（自然語言）+ 「方式 B：終端機一行」（備用指令）。教師永遠優先用自然語言與 AI 對話完成工作。

意義

v7 是系統從「個人工具」正式成為「組織基礎設施」的里程碑。多人協作、版本控制、權限管理、AI 口語操作——這些能力的組合，意味著一所華德福學校的教師團隊可以在同一個系統上協作，共享教育哲學的根基，同時保有各自的教學藝術空間。

更重要的是，教師從頭到尾不需要學任何程式語言。他們需要的只是：說出自己想做什麼，AI 負責翻譯成技術操作。這就是「意義架構師」的完整實現——教師定義意義，機器執行操作。

七次演化的洞見

從記憶到結構

v1 → v2 的跳躍不只是檔案數量增加，而是教師第一次以系統思維理解知識的層次：什麼是永恆的？什麼是變動的？什麼是當下的？這種分層思維本身就是華德福教育中「秩序帶來安全感」的數位延伸。

從產出到品質

v3 → v5 的演化揭示了一個關鍵洞見：AI 協作的價值不在於「產出更多」，而在於「每一份產出都照顧到每一個孩子」。差異化教學框架將 AI 從打字機升級為教學設計夥伴。品質標準則確保這個夥伴的表現穩定可靠。

從個人到集體

v6 → v7 的轉變是系統最深層的進化。當一個工具能服務整個教師團隊時，它就不再是工具——它是基礎設施。Workspace 隔離與 Git 協作的組合，創造了一個前所未有的可能：華德福教師團隊可以在保持個體教學自由的同時，共享智慧的基礎層。

從依賴到自主

貫穿七個版本的核心線索：教師越來越不依賴特定 AI 工具。技能正本是純文字檔，session.yaml 是開放格式，Git 是全球標準。如果某個 AI 明天消失，教師的知識體系、備課內容、教學框架全部安全地保存在自己的檔案系統中。這才是真正的數位自主權。

未來願景

TeacherOS 的下一步不是更多功能，而是更多教師。當第一位同事加入這個系統時，我們建設的一切將接受真實的考驗。

v8 預期：第一位同事加入

所有架構設計的真實壓力測試。Workspace 隔離是否真的有效？技能系統是否真的工具無關？教師手冊是否真的讓非技術教師順利上手？答案不在程式碼裡，在教室裡。

跨校擴展的可能

如果一所華德福學校的教師團隊能在這個系統上協作，那麼兩所呢？五所呢？台灣的華德福學校社群規模不大，但這恰恰是優勢——足夠小以維持緊密的教育理念共識，足夠大以驗證系統的可擴展性。

始終不變的核心

無論系統如何演化，有一件事永遠不會改變：最高度的技術自動化，是為了成就最純粹的低科技人文互動。教師走進教室的那一刻，放下所有技術，用溫暖的目光凝視每一個孩子。這就是 TeacherOS 存在的全部理由。

TeacherOS v7 Changelog — 2026-03-03

本版本涵蓋三輪架構重組的所有變更，是自 v6 以來最大的結構性更新。

版本摘要

v7 核心主題：多人協作基礎建設 + AI 口語操作介面

從單人系統升級為支援多位教師的協作系統，建立 Git 工作流程、存檔/合併技能、branch 自動化，並將教師手冊重寫為以 AI 口語指令為主的操作介面。

一、Git 多人協作模型

選定 Collaborator + Branch 模式（非 Fork 模式），確保教師能互相看到工作。Branch 命名規則：workspace/{Teacher_ID}。三層保護機制：Pre-commit Hook（軟）→ CODEOWNERS（硬）→ Branch Protection（硬）。Branch Protection 只保護 main 的合併，教師在個人 branch 上自由 commit。

二、新增技能：save（存檔）

觸發語：「存檔」「儲存」「幫我存」「commit」「上傳」「備份今天的工作」。流程：檢查變更 → 取得說明 → 執行 git add/commit/push → 確認結果。容錯：push 失敗時自動嘗試 git pull --rebase。

三、新增技能：pull-request（合併申請）

觸發語：「發 PR」「合併申請」「送回主系統」「通知 David 合併」。流程：檢查未存檔 → 收集 PR 資訊 → 確認 branch → 引導 GitHub PR 建立。安全：PR base 永遠是 main，只有 David 可合併。

四、新增腳本：save.sh（一行存檔）

路徑：setup/save.sh。用法：bash setup/save.sh "完成 Unit 2 教案"。功能：自動執行 git add → commit → push，帶彩色輸出。
